# Design Principles

These ten principles are the test every ETHDS design and implementation
decision is measured against. The full documentation for each — with
rationale, examples, anti-patterns, and implementation guidance — is in
the repository under
[`docs/design-principles/`](https://github.com/yidnekachewkibru/ethgov-design-system/tree/main/docs/design-principles).

| # | Principle | In one sentence |
|---|---|---|
| 1 | Start with Citizen Needs | Design decisions begin with what citizens need to accomplish, not how government is organised. |
| 2 | Multilingual by Default | Every component, pattern, and template supports all six languages from the start. |
| 3 | Mobile First | Design for small, constrained devices first, then enhance upward. |
| 4 | Accessibility by Default | WCAG 2.2 AA is the floor, not a final checklist. |
| 5 | Simple Before Powerful | The simplest solution that meets the need wins. |
| 6 | Design for Low Bandwidth | Services work on slow, metered, unreliable connections. |
| 7 | Reuse Before Building | Check ETHDS before creating something new. |
| 8 | Security and Privacy by Design | Citizen data protection is considered at every layer. |
| 9 | Consistent Government Experience | A citizen recognises and can use any government service. |
| 10 | Open by Default | Favour openness in process and code. |

## Why principles first

A service can be fast, accessible, and multilingual and still fail if it
forces a citizen to understand bureaucracy before they can act. The
principles exist so that when two good options conflict, there is a
shared, written basis for deciding — in design, in code review, and in
RFCs.
